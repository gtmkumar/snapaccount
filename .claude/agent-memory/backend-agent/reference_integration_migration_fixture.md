Integration suites use `tests/integration/_shared/MigrationSupport.cs` (namespace `SnapAccount.IntegrationTests.Shared`) instead of EF `EnsureCreated`. **Why:** EnsureCreated builds schema from the EF model, which diverges from the SQL migrations (the real schema source of truth) — it omits raw-SQL/reference tables (reporting tables, coa_template, loan_products), DB column defaults, CHECK constraints, native PG enums, and FKs the EF model doesn't declare. That made suites 500 on missing tables and hid EF↔DB divergences.

**How it works:** `MigratedPostgresFixture` (an xUnit `IAsyncLifetime` / `ICollectionFixture`) starts ONE `pgvector/pgvector:pg17` container per test assembly (pgvector needed — 000_init enables the `vector` extension), replays all `database/migrations/*.sql` in ordinal filename order (3-digit zero-padded, so lexical == version sort) ONCE into a template DB that MUST be named `snapaccount` (000_init hardcodes `ALTER DATABASE snapaccount SET search_path`). Each test gets an instant `CREATE DATABASE it_<guid> TEMPLATE snapaccount` clone (search_path re-applied on the clone; TEMPLATE doesn't copy ALTER DATABASE settings). Post-migrate the template is truncated of transactional + auth-config data (via `session_replication_role=replica` to bypass the accounting.edit_log append-only trigger) EXCEPT a keep-list of reference/template tables the app materializes from (coa_template, gst tax_rate/hsn_sac/notice_deadline_rules, itr tax_regime/tax_slab, loan_type, document_category, subscription_plan, shared config). Clones also run with `session_replication_role=replica` so FK/trigger enforcement is OFF on the seed path (tests self-seed partial graphs) — the code-under-test still sees the exact production SCHEMA. `MEMORY.md` design decision (b): FK-off on seed path, full real schema.

**Adopt in a suite:** (1) add to the .csproj `<Compile Include="..\_shared\MigrationSupport.cs" Link="_shared\MigrationSupport.cs" />`; (2) `using SnapAccount.IntegrationTests.Shared;`, class becomes `[Collection("migrated")] public class Foo(MigratedPostgresFixture pg) : IAsyncLifetime`; (3) `_connectionString = pg.NewDatabaseConnectionString()`, delete any per-class `PostgreSqlContainer`/`EnsureCreatedAsync`/`MigrateAsync`; (4) factory sets `UseSetting("ConnectionStrings:DefaultConnection", _connectionString)` + `UseSetting("DEV_AUTH_BYPASS","true")`; client sends `Bearer dev-superadmin-token`. AuthService/PostgresFixture.cs just inherits MigratedPostgresFixture (its `[Collection("integration")]` keeps working with zero per-test churn). A direct-DbContext test must replicate the composite's Npgsql `MapEnum<>` config (e.g. LoanService uses public `UpperSnakeCaseNameTranslator` for application_status_v2 + partner_bank_adapter_type) or native enum writes 500.

**Auth in tests:** org-scoped endpoints check `currentUser.OrganizationId == request org`. dev-superadmin-token → userId `22222222-2222-2222-2222-222222222222`, org `11111111-1111-1111-1111-111111111111` (FirebaseAuthMiddleware.DevAuthTokens). Use that org, not random GUIDs.

If a reference/config table an endpoint reads gets truncated, add it to the keep-list in MigrationSupport.cs (do NOT truncate it). Do NOT mask a real backend bug (native enum not registered in the COMPOSITE, EF mapping to nonexistent columns) with a test-side workaround — skip the test citing the bug so it stays visible. See [[project_integration_suite_migration_fixture_conversion]] and [[project_integration_suite_notif_sub_callback_chat_acct]] for the surfaced backend bugs.
