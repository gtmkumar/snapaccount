Bringing up the Android app for live E2E (AVD `snap_pixel`, emulator-5554) and the traps hit on 2026-07-05.

**Rule: this is a PLAIN React Native debug build — package.json has NO `expo-dev-client`/`expo-dev-launcher`.** So the `snapaccount://expo-development-client/?url=…` dev-client deep link does nothing; the app just connects to Metro at the default `10.0.2.2:8081`. Don't waste time on dev-client deep links or custom Metro ports — run Metro on **8081** and launch with `monkey -p com.snapaccount.app -c android.intent.category.LAUNCHER 1`.

**Env prerequisites (both were missing on this host):**
- No system Java. Set `JAVA_HOME=/Applications/Android Studio.app/Contents/jbr/Contents/Home` (JDK 21) for any Gradle/`expo run:android`.
- Gateway is on **:6060** here (macOS AirPlay squats :5000 → 403). App default is `http://localhost:5000` (app.json `extra.apiBaseUrl`); `src/lib/api.ts` rewrites localhost→10.0.2.2 for Android. Add `mobile/.env.local` with `EXPO_PUBLIC_API_BASE_URL=http://localhost:6060`. This is a local port override, NOT a product bug. See [[project_mobile_testing_state]].
- Watch for a stale Metro from the sibling `laundryghar/customer-mobile` project holding :8081 — kill it (or it serves the wrong bundle).

**Broken-APK signature (P7-QA-MOBILE-AND-BLOCK-01):** redbox "Unable to load script … index.android.bundle packaged correctly for release" with stack `loadJSBundleFromAssets → loadScriptFromAssets → JSBundleLoader.kt:33 → ReactHostImpl.kt:1059`, and Metro receives ZERO requests even though `10.0.2.2:8081` is reachable (test from device: `adb shell "… | toybox nc 10.0.2.2 8081"` → HTTP 200). Means the APK is wired to load JS from an embedded asset bundle (release JS mode) that isn't packaged — it will NEVER connect to Metro no matter what you do to the packager. Only a rebuild fixes it.

**Gradle-9 build trap (P7-QA-MOBILE-AND-BLOCK-02):** `npx expo run:android` fails fast with `Class org.gradle.jvm.toolchain.JvmVendorSpec does not have member field 'IBM_SEMERU'`. Cause: gitignored `mobile/android` prebuild ships Gradle wrapper **9.3.1** (removed `IBM_SEMERU` in Gradle 9.0) but the RN 0.85.3 / Expo ~56 Gradle plugin still references it. Fix is mobile-dev's: `expo prebuild --clean -p android` to regenerate a matching wrapper, or pin wrapper to Gradle 8.x. `mobile/android` is mobile-dev-owned (gitignored artifact) — escalate, don't edit it.

**Why:** on 2026-07-05 the full Android E2E sweep was fully BLOCKED at app launch by these two — no auth/business/individual flows could run.
**How to apply:** next Android sweep, first confirm a WORKING debug APK connects to Metro (watch for a bundle request in the Metro log within seconds of launch). If you see the loads-from-assets redbox, stop and get mobile-dev to rebuild — don't thrash on packager config.

## Live-driving quirks (2026-07-05 Android sweep)
- **Form fields: use Appium, not coordinate taps.** mobile-mcp `mobile_click_on_screen_at_coordinates` and raw `adb input tap` do NOT reliably focus a RN TextInput while the soft keyboard is already up (the app's ScrollViews use `keyboardShouldPersistTaps="handled"`; the keyboard also reflows the layout under ADJUST_RESIZE, so cached coordinates land on the wrong field). Create an Appium UiAutomator2 session (noReset, dontStopAppOnReset, autoLaunch:false) and use `appium_find_element` + `appium_set_value` / `appium_click`. Bottom-tab nav via Appium `descriptionContains("GST"/"Home")` or `//*[@text='...']` is reliable; coordinate taps on tabs get eaten by the dev LogBox toast overlapping the tab bar (dismiss it first).
- **RN surface goes BLACK on the GST and Loans dashboard tabs (AND-LIVE-01).** The app's RN surface stops compositing to the framebuffer (screenshot = solid black) specifically when navigating to the data-heavy gauge/chart dashboards — reproduced immediately on the first GST tap after a fresh login, and again on LoanHub. Home/Documents/onboarding-wizard (list/form screens) render fine. Android launcher + Appium page source both work → it's an app RN-surface/GL-context loss (candidate: a shared SVG/GL gauge/chart component), NOT an emulator GPU failure. Persists to all later screens until app RESTART. Before filing a "black screen" bug, ALWAYS check Appium `get_page_source` — content is present. Verify on a physical device / release build before treating as a release-blocker.
- **DO NOT derive bugs from the a11y tree while the surface is BLACK — it is STALE (CRITICAL, learned the hard way 2026-07-05).** When AND-LIVE-01 blacks the RN surface, `appium_get_page_source` still returns a tree, but that tree reflects a PRE-RENDER / partial state: unresolved i18n keys (showed raw `mobile.loan.hub.card.bank` that actually resolves to "Lending partner"), default/stale values ("May not qualify" vs the real "Not checked", "SnapAccount User" vs the real onboarding name "Rajesh Kumar Sharma"). I filed AND-LIVE-03/04/06 off black-surface captures and ALL THREE were false positives — retracted after the surface recovered and re-rendered. Navigation via the tree still works while black (for reaching a screen), but to CONFIRM any content/label/empty-state bug you MUST restart the app to recover the surface and re-verify visually. Only AND-LIVE-05 (consent 0/0 post-onboarding) survived re-verification because it matched a known real backend gap. `appium_get_page_source` is fine when the surface renders; avoid `adb shell uiautomator dump` while an Appium session is active (they contend for the uiautomator service and the adb dump hangs).
- **AND-LIVE-01 is INTERMITTENT, not deterministic.** It reproduced in a long/accumulated session but did NOT recur on a clean app restart across a full GST/Loans/More/DPDP sweep (logcat showed no app EGL/Reanimated/OOM crash). Treat black screens as a session-state/resource symptom: RESTART THE APP FIRST, then re-test — don't file content bugs off the black state.
- Dev LogBox warning toast ("Open debugger to view warnings") keeps reappearing and OVERLAPS the bottom tab bar — dismiss it (tap its ✕ ~device 995,2147) before tapping Loans/More/GST tabs, or the tap hits the toast.
- **OTP:** auto-submits on the 6th digit; a fast 6-char burst only registers the last 3 digits — enter digit-by-digit via the keypad or use Appium set_value. Dev OTP is `123456` for any number.
- **Re-login** with the same number after logout lands straight on Home (onboarding persists server-side; Home greeting switches from the person's name to the business name).
- **Compliance validators confirmed live:** PAN char-by-char (XXXXX9999X), GSTIN 15-char (`27AABCU9603R1ZM`), Aadhaar 12-digit Verhoeff (`999999990019`) + masked input.
