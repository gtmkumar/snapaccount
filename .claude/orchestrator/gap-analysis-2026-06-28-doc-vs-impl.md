> Branch: `feature/repository-refactor` (12→3 composite consolidation). Generated by a 13-area parallel doc↔code audit with per-gap adversarial verification (the prior gap docs had false positives, so every claim here was re-checked against the actual code).

## Cross-cutting themes

1. **Frontend/mobile ↔ backend contract divergence is the #1 issue.** Many admin & mobile screens call endpoints that don't exist (404) or parse responses whose shape/casing fails their Zod schema (loan-ops actions, ITR compute/draft, subscription self-service, dashboard KPIs, reports list, notification center, ARN/audit). These are silent runtime breakages.
2. **Orphaned engines / un-wired pipelines.** Code exists but nothing calls it: GSTR-3B calc service, the live Razorpay client, the notification fan-out (most I2 events have no subscriber), the in-app inbox channel, semantic RAG (query embedding computed then discarded).
3. **Real security/compliance holes.** RLS read-isolation is non-functional outside Auth (the `app.current_user_id` GUC is never set → the app connects as superuser); organisation business PAN is stored in plaintext; the Document module has no DPDP erasure subscriber.
4. **Infra provisioning lags the code.** The API gateway has no prod build/deploy path; DPDP account-deletion + GST/ITR/Callback recurring-job Pub/Sub subscriptions are never provisioned, so several wired backend handlers never run in prod.
5. **Mobile dead-ends.** Device binding (B1.3) is never triggered after login; Android SMS auto-read (B1.2) absent; no Appearance/theme screen; several i18n key-prefix bugs render raw keys.

- **Severity / Status:** high · partial · surface=`mobile` · owner=`mobile-dev`

- **Verified evidence:** Backend AddDevice path is real and complete: backend/Services/PlatformService/Platform.Application/Auth/Devices/Commands/AddDevice/AddDeviceCommand.cs is the ONLY caller of User.AddDevice() (verified via grep '\.AddDevice(' across Platform.Application — single hit).
